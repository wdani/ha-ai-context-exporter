# HA AI Context Exporter – Entwicklungs-Tree & Roadmap\n\n## Zweck dieser Datei\n\nDiese Datei ist die gemeinsame Orientierung für die weitere Entwicklung.\n\nSie kombiniert:\n\n1. **Langfristiges Zielbild**\n   - Was soll der HA AI Context Exporter später können?\n\n2. **Schritt-für-Schritt-Roadmap**\n   - Was wurde bereits gebaut?\n   - Was kommt als Nächstes?\n   - Welche Reihenfolge ist sinnvoll?\n\nWichtig:\nDas Projekt soll **nicht lose** wachsen.\nEs soll entlang einer **klaren Linie** entwickelt werden, damit es am Ende deinen Anforderungen entspricht:\n\n- möglichst vollständiges Bild von Home Assistant\n- saubere, ehrliche, brauchbare Daten\n- Verknüpfung von API-Zustand und YAML-/Dateikontext\n- Beziehungen zwischen Bereichen, Geräten, Entitäten, Integrationen, Automationen und Dashboards\n- Export, den eine KI wirklich nutzen kann, um:\n  - das System zu verstehen\n  - bestehende Automationen zu prüfen\n  - neue Automationen vorzuschlagen\n  - Benennung und Struktur zu verbessern\n- optional auch interne Analysefunktionen direkt im Tool\n\n---\n\n# 1. Leitbild des Projekts\n\nDer **HA AI Context Exporter** soll langfristig **beides** sein:\n\n## A. Technischer Exporter\nEin Tool, das Home Assistant-Daten strukturiert, sicher und möglichst vollständig exportieren kann.\n\n## B. Analyzer / Verständnisschicht\nEin Tool, das Zusammenhänge sichtbar macht und eine KI in die Lage versetzt,\nsich **ein komplettes Bild des Home Assistant Systems** zu machen.\n\nDas bedeutet später nicht nur:\n- Wie viele Entities gibt es?\n- Wie viele Automationen gibt es?\n\nSondern auch:\n- Welche Entity gehört zu welchem Device?\n- Welches Device gehört zu welchem Bereich?\n- Welche Integration steckt dahinter?\n- Welche Automationen referenzieren welche Entities?\n- Welche Dashboards zeigen welche Entities?\n- Welche YAML-/Dateidefinitionen erzeugen welches Verhalten?\n\n---\n\n# 2. Grundsatzentscheidungen\n\n## 2.1 Entity-Daten möglichst vollständig erfassen\n\nSpäter sollen Entities möglichst vollständig erfasst werden.\n\nDazu gehören z. B.:\n- entity_id\n- domain\n- state\n- friendly_name\n- area\n- device\n- integration\n- attributes\n\nAttribute werden später in zwei Ebenen aufgeteilt:\n\n### raw_attributes\n- vollständige Originalattribute\n- technisch vollständig\n- eher für tiefe Analyse / Vollkontext\n\n### important_attributes\n- reduzierte, sinnvolle Teilmenge\n- KI-relevante Attribute\n- besser lesbar und kompakter\n\nWichtig:\n**Erst Datenreichtum schaffen, danach über Varianten filtern.**\n\n---\n\n## 2.2 YAML / Dateikontext aktiv mit API-Zustand verknüpfen\n\nDateikontext ist **nicht nur nice-to-have**, sondern ein echter Zielbestandteil.\n\nSpäter soll das Tool nicht nur:\n- API-Daten lesen\n\nsondern auch:\n- YAML-Dateien lesen\n- Konfiguration verstehen\n- Beziehungen zwischen Dateidefinition und laufendem Zustand herstellen\n\nBeispiele:\n- Automation aus `automations.yaml`\n- referenzierte Entities aus API\n- Area / Device / Integration Kontext dazu\n- später evtl. Inkonsistenzen und Verbesserungshinweise\n\nWichtig:\nDas Ziel ist **nicht** API und Dateien nebeneinander zu exportieren,\nsondern **beides aktiv zu verknüpfen**.\n\n---\n\n# 3. Entwicklungs-Tree (Zielbild)\n\n## 3.1 Foundation Layer\nTechnische Basis des Add-ons.\n\nEnthält:\n- Add-on Scaffold\n- Ingress UI\n- Python Backend\n- Health / Info / Preview Endpoints\n- Core Proxy Zugriff\n- read-only Sicherheitsmodell\n- Export Controller\n- Download-System\n- AI-Projektdokumentation\n- Review-/Changelog-/Workflow-Struktur\n\nZiel:\nEine stabile, sichere und erweiterbare Plattform.\n\nStatus:\n**fertig**\n\n---\n\n## 3.2 Discovery Summary Layer\nErste Überblicksebene.\n\nEnthält:\n- system summary\n- entities summary\n- logic summary\n- dashboard summary\n- integrations summary\n\nZiel:\nEin schneller, kompakter Überblick über das HA-System.\n\nWichtig:\nDiese Ebene ist wichtig, aber **nicht das Endziel**.\n\nStatus:\n**größtenteils vorhanden**\n\n---\n\n## 3.3 Discovery Quality Layer\nQualitätsverbesserung der vorhandenen Discovery-Daten.\n\nEnthält:\n- Integrationsbereinigung\n- Entschärfung interner/Core-Komponenten\n- saubere Status-Semantik\n- bessere Areas-/Devices-Erkennung\n- robustere Dashboard-Erkennung\n\nZiel:\nVor neuen Tiefenebenen zuerst die vorhandenen Daten sauber machen.\n\nStatus:\n**nächste aktive Phase**\n\n---\n\n## 3.4 Entity Context Layer\nTiefe Informationen über Entities.\n\nSpäter pro Entity z. B.:\n- entity_id\n- domain\n- state\n- friendly_name\n- integration\n- device\n- area\n- raw_attributes\n- important_attributes\n\nZiel:\nEntities nicht nur zählen, sondern **kontextualisieren**.\n\nStatus:\n**später, aber sehr wichtig**\n\n---\n\n## 3.5 Relationship Layer\nSystembeziehungen modellieren.\n\nBeispiele:\n- Entity → Device\n- Device → Area\n- Entity → Integration\n- Automation → Entity\n- Dashboard → Entity\n- Bereich → Devices / Entities / Automationen\n\nZiel:\nDie KI soll nicht nur Dinge sehen, sondern verstehen,\n**wie die Dinge zusammenhängen**.\n\nStatus:\n**später, nach Entity Context**\n\n---\n\n## 3.6 Logic Deep Layer\nTiefe Sicht auf Automationen, Scripts und Szenen.\n\nSpäter z. B.:\n- alias\n- triggers\n- conditions\n- actions\n- referenced_entities\n- referenced_areas\n- referenced_devices\n\nZiel:\nDie KI soll bestehende Logik prüfen und verbessern können.\n\nStatus:\n**später**\n\n---\n\n## 3.7 Dashboard Deep Layer\nTiefe Sicht auf Dashboards / Lovelace.\n\nSpäter z. B.:\n- dashboard names\n- views\n- cards\n- card types\n- referenced_entities\n- Bereichs-/Themenbezug\n\nZiel:\nDie UI-Sicht von Home Assistant wird KI-lesbar.\n\nStatus:\n**später**\n\n---\n\n## 3.8 File / YAML Context Layer\nDatei- und Konfigurationssicht.\n\nSpäter:\n- `automations.yaml`\n- `scripts.yaml`\n- `scenes.yaml`\n- `configuration.yaml`\n- packages / includes\n\nZiel:\nDie KI sieht nicht nur API-Zustand, sondern auch die eigentliche Definition.\n\nStatus:\n**später, aber ausdrücklich gewünscht**\n\n---\n\n## 3.9 LLM Optimization Layer\nExport speziell für KI optimieren.\n\nSpäter:\n- Variant A = technische Vollversion\n- Variant B = kompakter / KI-orientierter\n- Zusammenfassungen\n- Noise Reduction\n- Priorisierung\n\nZiel:\nNicht nur viele Daten exportieren, sondern **nutzbaren KI-Kontext** erzeugen.\n\nStatus:\n**später**\n\n---\n\n## 3.10 Internal Analyzer Layer\nDas Tool selbst liefert Hinweise.\n\nSpäter z. B.:\n- Benennungsqualität von Entities / Helpers\n- Inkonsistenzen in Struktur\n- redundante Automationen\n- auffällige Konfigurationsmuster\n- Verbesserungsvorschläge\n\nZiel:\nDas Tool wird nicht nur Exporter, sondern auch aktiver Helfer.\n\nStatus:\n**später / Ausbauphase**\n\n---\n\n# 4. Schritt-für-Schritt-Roadmap\n\n## Phase 0 – Fundament bauen\n### Ziel\nDas Projekt überhaupt lauffähig, sicher und strukturiert machen.\n\n### Enthielt\n- Add-on Scaffold\n- Ingress UI\n- Backend HTTP Server\n- erste Preview-/Discovery-Endpunkte\n- `/api/export`\n- `/api/export/download`\n- Download Header\n- Markdown Renderer\n- Versioning-Grundlage\n- Export-Semantik\n- Core Proxy Zugriff\n- `/api/ha-auth-debug`\n- AI-Projektdokumentation\n- Review-/Changelog-Workflow\n\n### Status\n**abgeschlossen**\n\n---\n\n## Phase 1 – Erste echte Discovery\n### Ziel\nNicht nur Struktur, sondern reale Daten liefern.\n\n### Enthielt\n- entities discovery\n- top domains\n- logic counts\n- erste integrations discovery\n- system partial summary\n- saubere Status-Semantik\n\n### Status\n**laufend, aber wesentliche Basis steht**\n\n---\n\n## Phase 2 – Discovery bereinigen\n### Ziel\nDie aktuell vorhandenen Daten sauberer und sinnvoller machen, bevor wir zu viel Neues aufbauen.\n\n### Warum diese Phase wichtig ist\nDein Wunsch ist:\n**erst saubere, brauchbare Daten, dann Erweiterung**.\n\n### Geplante Schritte\n1. Integrationsliste bereinigen\n   - Plattformen wie `mqtt.sensor`, `mqtt.switch` sinnvoll auf Hauptintegration zurückführen\n   - interne/Core-Komponenten entschärfen oder markieren\n\n2. Areas / Devices Discovery verbessern\n   - prüfen, ob / wie die Endpunkte robuster nutzbar werden\n   - ehrliche Semantik beibehalten\n\n3. Dashboard Discovery verbessern\n   - robustere Lesbarkeit\n   - bessere Metadaten\n\n### Status\n**jetzt aktuelle Hauptphase**\n\n---\n\n## Phase 3 – Entity Context einführen\n### Ziel\nVon Counts zu echten Entitätskontexten wechseln.\n\n### Geplante Schritte\n1. Erste kompakte und vollständige Entity-Struktur definieren\n2. raw_attributes exportieren\n3. important_attributes definieren\n4. Integration / Device / Area Kontext ergänzen\n\n### Ergebnis\nDie KI kann einzelne Entities verstehen, nicht nur Mengen.\n\n### Status\n**nach Phase 2**\n\n---\n\n## Phase 4 – Relationship Layer aufbauen\n### Ziel\nZusammenhänge im System modellieren.\n\n### Geplante Schritte\n1. Entity → Device\n2. Device → Area\n3. Entity → Integration\n4. Automation → Entities\n5. Dashboard → Entities\n\n### Ergebnis\nDer Export wird von einer Sammlung zu einem **Systemmodell**.\n\n### Status\n**nach Entity Context**\n\n---\n\n## Phase 5 – Logic Deep Export\n### Ziel\nAutomationen, Scripts und Szenen wirklich verständlich exportieren.\n\n### Geplante Schritte\n1. Automation-Struktur\n   - alias\n   - triggers\n   - conditions\n   - actions\n   - referenced_entities\n\n2. Scripts vertiefen\n3. Szenen vertiefen\n\n### Ergebnis\nDie KI kann bestehende Logik prüfen und neue Logik sinnvoll darauf aufbauen.\n\n### Status\n**nach Relationship Layer**\n\n---\n\n## Phase 6 – Dashboard Deep Export\n### Ziel\nLovelace/Dashboard-Struktur tiefer erfassen.\n\n### Geplante Schritte\n1. Dashboard-Namen / Views / Karten\n2. Karten-Typen\n3. referenzierte Entities\n4. Bereichs-/Themenbezug\n\n### Ergebnis\nDie UI-Sicht von Home Assistant wird KI-lesbar.\n\n### Status\n**später, aber wichtig**\n\n---\n\n## Phase 7 – File / YAML Context\n### Ziel\nDas echte Konfigurationsbild ergänzen.\n\n### Geplante Schritte\n1. Dateizugriff / Lesestrategie klären\n2. `automations.yaml`\n3. `scripts.yaml`\n4. `scenes.yaml`\n5. `configuration.yaml`\n6. packages / includes\n7. aktive Verknüpfung mit API-Zustand\n\n### Ergebnis\nDie KI sieht nicht nur API-Zustände, sondern die eigentlichen Definitionsdateien und ihre Beziehungen.\n\n### Status\n**später, aber ausdrücklich gewünscht und wichtig**\n\n---\n\n## Phase 8 – LLM-Optimierung\n### Ziel\nDen Export so formen, dass KI maximal sinnvoll damit arbeiten kann.\n\n### Geplante Schritte\n1. Varianten klar schärfen\n2. kompakte Zusammenfassungen\n3. Noise reduzieren\n4. Priorisierung relevanter Informationen\n\n### Ergebnis\nNoch bessere Nutzbarkeit für ChatGPT, Claude, lokale LLMs usw.\n\n### Status\n**nach mehr inhaltlicher Tiefe**\n\n---\n\n## Phase 9 – Internal Analyzer\n### Ziel\nDas Tool selbst gibt schon Hinweise.\n\n### Beispiele\n- Benennungsqualität\n- unklare Entities / Helpers\n- Strukturprobleme\n- auffällige oder redundante Automationen\n- organisatorische Verbesserungsvorschläge\n\n### Ergebnis\nDas Tool wird nicht nur Exporter, sondern aktiver Helfer.\n\n### Status\n**später / Ausbauphase**\n\n---\n\n# 5. Varianten- und Modus-Idee (frühe Leitlinie)\n\nDu hast dich für **B = mehrere Modi** entschieden.\nDas ist sinnvoll, weil vollständige Daten später sehr groß werden können.\n\n## compact\nZiel:\nKompakter Überblick.\n\nEnthält später vor allem:\n- counts\n- top lists\n- kurze Zusammenfassungen\n- stark reduzierte Entity-/Integrationsinformationen\n\nFür:\n- schnelle Übersicht\n- kleine Exporte\n- erste KI-Einschätzung\n\n## standard\nZiel:\nGute Balance zwischen Vollständigkeit und Lesbarkeit.\n\nEnthält später:\n- Summary + wichtige Beziehungen\n- kompakte Entity-Daten\n- important_attributes\n- ausgewählte Logik-/Dashboard-/Integrationskontexte\n\nFür:\n- normale Analyse\n- Standard-KI-Exports\n\n## full\nZiel:\nMöglichst vollständiger technischer Export.\n\nEnthält später:\n- vollständige Entities\n- raw_attributes + important_attributes\n- mehr Beziehungsdaten\n- tieferen Logic-/Dashboard-/Dateikontext\n\nFür:\n- tiefe Analyse\n- Debugging\n- vollständiger Systemkontext\n\n## analysis\nZiel:\nNicht primär Rohdaten, sondern Analyse- und Verbesserungsfokus.\n\nEnthält später:\n- Auffälligkeiten\n- Strukturprobleme\n- Benennungsprobleme\n- Redundanzen\n- mögliche Verbesserungsfelder\n- zusammengefasste Erkenntnisse\n\nFür:\n- Optimierung\n- Review\n- Refactoring-Unterstützung\n- KI-Beratung\n\nWichtig:\nDiese Modus-Beschreibungen sind vorerst eine **Leitlinie**.\nDie konkrete technische Befüllung kann später schrittweise präzisiert werden.\n\n---\n\n# 6.

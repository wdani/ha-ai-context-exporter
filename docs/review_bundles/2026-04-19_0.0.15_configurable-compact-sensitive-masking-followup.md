## Summary

- Added the add-on option `allow_sensitive_values`, defaulting to `false`; only explicit boolean `true` allows compact sensitive values to remain unmasked.
- Kept missing or invalid option values privacy-safer by falling back to masking enabled.
- Added concise user-facing best-effort masking warnings to the export `warnings` list, Markdown rendering, the existing export UI surface, and README.
- Did not change compact item structure, export key names, status fields, category names, sorting semantics, or the `important_attributes` whitelist.
- Did not introduce `raw_attributes`, relationship modeling, new export sections, dashboard/areas/devices/integrations logic, YAML/file parsing, WebSocket access, POST/template fallback, service calls, state-changing behavior, or a full privacy redesign.
- Bumped both `ha_ai_context_exporter/rootfs/app/version.py` and `ha_ai_context_exporter/config.yaml` from `0.0.14` to `0.0.15`.

## Remaining limitations

- Masking is best-effort only and does not claim complete anonymization.
- Arbitrary names, addresses, network names, device identifiers, and other sensitive values can still be missed if they do not match the current evidenced compact patterns.
- The scope remains limited to already exported compact string-bearing entity fields: `entity_id`, `state`, `friendly_name`, and string values inside `important_attributes`.

# Review Bundle

## Summary

- Treated this as the follow-up task `stabilize-ingress-active-mode-and-promote-transition-banner`.
- Inspected the current 0.0.19 merged ingress masking-status implementation before changing code.
- Confirmed the merged baseline already had a persistent active-mode block, a separate temporary transition-status block, frontend runtime memory for the previous active mode, and read-only `/api/info` polling.
- Identified the remaining UX gap: the active-mode block still carried strong whole-block mode coloring, while the reconnect/recovered block sat as a peer below it, so the experience could still feel like the same status card swapping emphasis.
- Made the active-mode block visually calmer and source-of-truth-like by using neutral card styling plus a small mode badge instead of full-block mode coloring.
- Promoted the temporary reconnect/recovered lifecycle into a stronger banner above the active-mode block so it becomes the primary visible event signal.
- Added an explicit `RECOVERED_STATUS_MIN_DWELL_MS = 15000` dwell rule so the reachable-again / now-active confirmation remains visible before the next successful `/api/info` refresh can return the page to normal steady state.
- Kept the current active mode visible independently from transition messaging; when the backend is temporarily unavailable, it still shows the last confirmed mode instead of pretending to have fresh data.
- Kept the implementation frontend-only with no backend status payload change, export payload behavior change, masking behavior change, export category change, or warning structure change.
- Did not add a writable settings UI, config mutation endpoint, Supervisor restart control logic, POST/PUT/PATCH/DELETE behavior, service calls, WebSocket access, raw attributes, relationship modeling, export redesign, or persistence beyond page runtime memory.
- Bumped both `ha_ai_context_exporter/rootfs/app/version.py` and `ha_ai_context_exporter/config.yaml` from `0.0.19` to `0.0.20` because this step adds a real user-visible UX hierarchy improvement.

## Remaining limitations

- This UX change cannot prevent, suppress, or replace Home Assistant Supervisor-level restart conflict messages such as another restart job already running.
- The export page only displays and refreshes the current add-on option state; users still change `allow_sensitive_values` in the Home Assistant add-on configuration.
- Masking remains best-effort only and users must manually review exports before sharing them.
- No browser runtime or Home Assistant ingress restart test was run in this local environment.
